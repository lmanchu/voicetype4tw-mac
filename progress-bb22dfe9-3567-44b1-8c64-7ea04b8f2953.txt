## Codebase Patterns

### STT Engine Implementation Pattern
1. All STT engines inherit from `stt/base.py:BaseSTT`
2. Required method: `transcribe(self, audio_bytes: bytes, language: str = "zh") -> str`
3. Audio format: WAV bytes (16-bit, mono, 16kHz sample rate)
4. Lazy loading: MLX-based models can be lazy-loaded in `__init__` or on first use

### Audio Conversion Pattern
```python
# WAV bytes → float32 numpy array [-1, 1]
import io
import wave
import numpy as np

audio_io = io.BytesIO(audio_bytes)
with wave.open(audio_io, "rb") as wf:
    raw_data = wf.readframes(wf.getnframes())
    audio_np = np.frombuffer(raw_data, dtype=np.int16).astype(np.float32) / 32768.0
```

### MLX Engine Pattern
- Import MLX library in transcribe() method (lazy load)
- Model loaded in `__init__` with repo path
- Pass model to transcribe function

### __init__.py Pattern
- Export all STT engines in `stt/__init__.py`
- Use relative imports: `from .qwen3_asr import Qwen3ASRSTT`

## Notes

### Audio Format Issue
The task specification says to use:
```python
audio_np = np.frombuffer(audio_bytes, dtype=np.int16).astype(np.float32) / 32768.0
```

However, `audio_bytes` is WAV format (from `AudioRecorder.stop()`), not raw int16. This will cause the WAV header to be interpreted as audio data. Need to verify if Qwen3-ASR can handle WAV directly or if proper WAV parsing is needed.
